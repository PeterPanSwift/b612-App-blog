一步一腳印，修正 Xcode App 專案的版本管理衝突(conflict)

Git 版本管理大部分的時候都很聰明，儘管我們的 App 是多人一起開發，它還是可以將大家的修改自動合併，不用我們操心。

But，人生最怕聽到這個 but，偶爾我們還是會不幸遇到 Git 無法自動合併，需要我們手動修正的問題。接下來就讓我們親眼見證 Git 合併失手的悲傷時刻，以及一步步修正解決問題吧。

故事是這樣開始的，很久很久以前，彼得潘和虎克船長一起合作開發 Swift iOS App。

1 彼得潘建立 Xcode 專案 DemoApp，然後將它上傳到 GitHub。
2 虎克船長利用 git clone 取得專案 DemoApp。
3 虎克船長加了 ThirdViewController.swift，然後進行 commit & push。
4 彼得潘加了 SecondViewController.swift，然後進行 commit。

接著彼得潘要 push 時遇到問題，因為要先 pull 虎克船長上傳到 GitHub 的東西。

5 彼得潘進行 pull。

此時將顯示 pull 的視窗，可惜不幸的，右下角的 pull 竟然不能點選 !

不能 pull 的原因是因為 Git 發現有些檔案它無法自動合併，比方同時有兩個人修改檔案，它不知道如何將兩個人的修改合併。

有問題無法合併的檔案我們習慣說它們有衝突(conflict)，因此如下圖所示，Xcode 會在它身上標記可怕的紅色 C。

此時我們只能自力自強，靠人類聰明的大腦合併。

6 彼得潘針對有衝突的檔案進行人工修正。

找出標記 C 的檔案，尋找其中有問題的紅色區塊，一個一個修正。

我們發現有問題的檔案是 project.pbxproj。(此檔案在 xcodeproj 下，xcodeproj 其實是個資料夾）

為什麼此檔會有問題呢 ? 因為它記錄了專案的相關資訊，比方 Project Navigator 裡有哪些檔案。因為彼得潘加了 SecondViewController.swift，虎克船長加了 ThirdViewController.swift，於是 Git 就混亂了，它不懂合併後該兩個檔案都保留，還是只保留其中一個。

沒關係，Git 不懂，我們就幫幫它吧。

下圖的紅色區塊告訴我們檔案裡無法合併的區塊，左邊是 Mac 本機的版本(HEAD)，右邊是 GitHub 上的版本，因此左邊的文字提到彼得潘加入的 SecondViewController.swift，右邊的文字提到虎克船長加入的 ThirdViewController.swift。

要如何修正呢 ? 很簡單，點選中間的問號 icon，然後告訴 Git 我們的選擇。

總共有以下四種選擇:

1 Choose Left: 選左邊本機的版本。

2 Choose Right: 選右邊 GitHub 上的版本。

3 Choose Left Then Right: 兩個都要，合併時將右邊的文字擺到左邊文字的後面。

4 Choose Right Then Left: 兩個都要，合併時將左邊的文字擺到右邊文字的後面。

以目前的例子，我們希望同時保留 SecondViewController.swift & ThirdViewController.swift，因此點選 Choose Left Then Right。(其實選 Choose Right Then Left 也可以)

如下圖所示，現在左邊畫面的文字同時包含 SecondViewController.swift & ThirdViewController.swift，中間的 icon 也不再顯示問號。

有衝突的地方往往不只一個，請有耐心地一個個解決。

我們可點選中間下方的 > 快速跳到下一個有問題的區塊。1/5 表示有 5 個區塊，目前我們在第 1 個區塊。

依序解決其它的紅色區塊，在此我們都選擇 Choose Left Then Right。

有些區塊如下圖所示，顯示為藍色。看到不是紅色的區塊真令人開心，因為它代表 Git 已幫我們合併，不用我們操心。

什麼時候才能 pull 呢 ?當每個衝突檔案的紅色區塊都被手動合併，不再顯示問號後，即是右下角的的 Pull 按鈕可點選之日。

7 彼得潘修正有衝突的檔案後，點選 pull。

彼得潘順利取得虎克船長上傳的 ThirdViewController.swift。

8 彼得潘點選 push。

將新增的 SecondViewController.swift 上傳到 GitHub。

9 虎克船長點選 pull。

虎克船長也順利取得彼得潘上傳的 SecondViewController.swift。
