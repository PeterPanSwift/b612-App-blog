利用 capture list 解決 Timer & NotificationCenter 造成的記憶體問題

開發 iOS App 時我們時常使用定時觸發的 timer & 接收通知的 notification center， 不過它們其實暗藏了一個可怕的陷阱，有可能讓 App 的記憶體愈用愈多，進而引發 App 因記憶體爆掉而閃退。

接下來讓我們透過實際的例子了解問題，然後討論解決的方法。

假設我們點選小人物的 button 進入第二頁的 BookViewController。

class BookViewController: UIViewController {
    
    deinit {
        print(#function)
    }

我們在 BookViewController 裡定義物件死掉時觸發的 deinit，如下圖所示，正常情況下，當我們進入 BookViewController 頁面，然後再回到首頁時，BookViewController 將順利死掉，印出 deinit。

不過當我們在 BookViewController 使用到以下的 Timer & NotificationCenter 寫法時，將造成 BookViewController 頁面被關閉後可怕地像幽靈般繼續活著。

  • scheduledTimer(timeInterval:target:selector:userInfo:repeats:)

以 scheduledTimer(timeInterval:target:selector:userInfo:repeats:) 產生 timer 時將 repeats 設為 true，造成 controller 的 reference count 增加，因此 controller 頁面被關閉後將繼續活著。

@objc func update() {
        
}
    
 override func viewDidLoad() {
     super.viewDidLoad()
     Timer.scheduledTimer(timeInterval: 1, target: self, selector: #selector(update), userInfo: nil, repeats: true)
        
}
  • scheduledTimer(withTimeInterval:repeats:block:)

在 scheduledTimer 的 closure 裡存取 self，造成 controller 的 reference count 增加，因此 controller 頁面被關閉後將繼續活著。

override func viewDidLoad() {
    super.viewDidLoad()
    Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { _ in
        self.view.backgroundColor = UIColor(red: .random(in: 0...1), green: 0, blue: 0, alpha: 1)
    }
        
}
  • addObserver(forName:object:queue:using:)

在 addObserver 的 closure 裡存取 self，造成 controller 的 reference count 增加，因此 controller 頁面被關閉後將繼續活著。

override func viewDidLoad() {
    super.viewDidLoad()
        
    NotificationCenter.default.addObserver(forName: UIResponder.keyboardDidShowNotification, object: nil, queue: .main) { (notification) in
        self.view.backgroundColor = UIColor(red: CGFloat.random(in: 0...1), green: 0, blue: 0, alpha: 1)
    }
}

解法:

方法1: 利用 capture list

以上程式造成問題的原因跟 closure 有關，因此不只是 Timer & NotificationCenter 會遇到 。當我們在 closure 裡存取物件時，它將增加物件的 reference count，因此有可能造成記憶體的問題。

此問題的解法是搭配 capture list，我們可在 closure 裡利用 [weak self] 讓 closure 別增加 controller 的 reference count，如此 controller 頁面關閉時它才能安心地死去，不再受 Timer & NotificationCenter 的羈絆而隱形地活著。

Timer

Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { [weak self] _ in
    guard let self else { return }
    view.backgroundColor = UIColor(red: .random(in: 0...1), green: 0, blue: 0, alpha: 1)
}

加了 [weak self] 後，controller 頁面關閉時可以順利死掉，不過 timer 依然會繼續運作，每秒執行 closure 的程式。因此記得在發現 controller 已經死掉，self 是 nil 時呼叫 timer 的 invalidate 停止 timer 的運作。

Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { [weak self] timer in
     guard let self else {
        timer.invalidate()
        return
     }
     view.backgroundColor = UIColor(red: .random(in: 0...1), green: 0, blue: 0, alpha: 1)
        
}

方法2: 呼叫 invalidate 停止 timer

解決 timer 問題的關鍵其實是 invalidate，它將讓 timer 終止，不再觸發 closure。所以我們也可以採用以下解法，在 controller 畫面消失時呼叫 invalidate。採用此法的話，在 timer 的 closure 裡不加 capture list 也是可以的。

var timer: Timer?
    
override func viewDidLoad() {
    super.viewDidLoad()
        
    timer = Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { timer in
        self.view.backgroundColor = UIColor(red: .random(in: 0...1), green: 0, blue: 0, alpha: 1)
    }
        
}
    
override func viewDidDisappear(_ animated: Bool) {
    super.viewDidDisappear(animated)
    timer?.invalidate()
}

方法3: 使用 combine 自動會停止的 timer publisher

iOS 13 的 Combine 讓 timer 也能成為發送 value 的 publisher，透過它我們的 timer 程式可以更簡單，而且當 controller 關閉時，timer 也會自動停止，不用另外呼叫 invalidate。有興趣的可參考以下連結。
