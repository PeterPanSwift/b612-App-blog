開發 SwiftUI App 時，我們常串接多個 modifier 調整元件的樣式跟功能。大部分的 modifier 可以用任意順序串接，然而某些 modifier 卻要特別注意順序問題，以下我們分幾種情況討論。

SwiftUI modifier 的順序(order)

開發 SwiftUI App 時，我們常串接多個 modifier 調整元件的樣式跟功能。大部分的 modifier 可以用任意順序串接，然而某些 modifier 卻要特別注意順序問題，以下我們分幾種情況討論。

  • modifier 的順序不影響結果。
  • modifier 的順序影響結果。
  • modifier 的順序影響可呼叫的 modifier。

modifier 的順序不影響結果

有些 modifier 的順序不影響結果，比方以下的 foregroundStyle ＆ background。

先設定前景再設定背景，或是先設定背景再設定前景，結果都是黃色背景的藍色文字。

struct ContentView: View {
    var body: some View {
        Text("Peter Pan")
            .foregroundStyle(.blue)
            .background(.yellow)
    }
}
struct ContentView: View {
    var body: some View {
        Text("Peter Pan")
            .background(.yellow)
            .foregroundStyle(.blue)
    }
}

modifier 的順序影響結果

有些 modifier 的順序會影響結果，比方以下例子的 padding & background。

先呼叫 padding 加上邊距，再呼叫 background 設定背景，因此文字跟邊框間的邊距也會填滿黃色背景。

struct ContentView: View {
    var body: some View {
        Text("Peter Pan")
            .padding(20)
            .background(.yellow)
            .border(.black, width: 5)
    }
}

先呼叫 background 設定背景，再呼叫 padding 加上邊距，因此文字跟邊框間的邊距沒有填色。

struct ContentView: View {
    var body: some View {
        Text("Peter Pan")
            .background(.yellow)
            .padding(20)
            .border(.black, width: 5)
    }
}

modifier 的順序影響可呼叫的 modifier

大部分的 modifier 可從任何 view 呼叫，不過有些 modifier 只能從特定型別呼叫，因此順序將造成某些 modifier 不能呼叫。

比方 resizable 只能從 Image 型別的資料呼叫，因此以下寫法是合法的。

struct ContentView: View {
    var body: some View {
        Image(systemName: "swift")
            .resizable()
            .scaledToFit()
    }
}

然而以下寫法會出現錯誤 Value of type 'some View' has no member 'resizable'。

scaledToFit 回傳的資料型別是 some View，不再是 Image，因此無法呼叫專屬於 Image 的 resizable。
