處理未來 case 的 @unknown default — Swift 5

Swift 5 SE-0192 的 Handling Future Enum Cases 加了處理未來 case 的 @unknown default。不是原本就有 default ，為什麼還要 @unknown default 呢 ?

讓我們看看以下例子，瞧瞧 @unknown default 的功用。

switch UIApplication.shared.statusBarOrientation {
case .portrait:
   print("portrait")
case .portraitUpsideDown:
   print("portraitUpsideDown")
case .landscapeLeft:
   print("landscapeLeft")
case .landscapeRight:
   print("landscapeRight")
case .unknown:
   print("unknown")
}

以上的程式利用 switch case 檢查 status bar 的方向。statusBarOrientation 的型別是 UIInterfaceOrientation，我們已將它底下 5 種 case 全都列出，所以這樣的程式應該是萬無一失，十分完美的。

然而在 Swift 5，它卻出現了黃色警告。

Switch covers known cases, but 'UIInterfaceOrientation' may have additional unknown values, possibly added in future versions
Handle unknown values using "@unknown default"

警告說我們的 switch 已經包含 UIInterfaceOrientation 所有的 case，但是它可能在未來增添新的 case，比方在 iOS 99 加入新的 case ，因此我們最好加上 @unknown default。

switch UIApplication.shared.statusBarOrientation {
case .portrait:
   print("portrait")
case .portraitUpsideDown:
   print("portraitUpsideDown")
case .landscapeLeft:
   print("landscapeLeft")
case .landscapeRight:
   print("landscapeRight")
case .unknown:
   print("unknown")
@unknown default:
   print("unknown default")
}

有了 @unknown default ，我們再也不怕未來 UIInterfaceOrientation 增添新的 case。將來倘若有新的 case，switch 的程式將執行到 @unknown default，我們的程式仍然可以正常運作。而且十分貼心的，當有新的 case 產生時，它還會警告我們，問我們是否要加入新的 case。

比方以下例子，我們故意漏寫 landscapeRight，假裝 landscapeRight 是新發明的 case，於是 Xcode 馬上顯示貼心的黃色警告訊息，Switch must be exhaustive，Do you want to add missing cases?

相反的，從前沒有 @unknown default 時，若是 UIInterfaceOrientation 增加新的 case，當初我們包含 enum 每個 case 的 switch 程式將產生 compile 錯誤，因為它不再完美，沒有包含所有的 case。

所以什麼時候該用 default，什麼時候該用 @unknown default 呢 ?

  1. 當你的 switch 已經列出 enum 的所有 case，此時則適合用 @unknown default，預防將來有新的 case 產生。
  2. 若你的 switch 只想處理其中幾種 case，不想列出所有 case，則適合搭配 default，讓沒列出的 case 都執行 default 下的程式。

最後有一點小細節值得注意的，@unknown default 只適用於 c 定義的 enum，所以我們用 swift 定義的 enum 是不會搭配 @unknown default 的。
