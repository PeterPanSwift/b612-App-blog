table 的 estimatedRowHeight UITableViewAutomaticDimension 問題

現在 iOS 的表格預設都是自動計算 cell 高度，比方點選 Storyboard 裡的 Table View 元件，我們可在 Size Inspector 發現它的 Row Height & Estimate 的 Automatic 被勾選，都是自動計算高度。

不過當我們啟動 App 後，將發現一個很特別的現象。比方如下圖所示，我們一開始有 10 筆資料，畫面一開始只看到兩個 cell。

此時 function tableView(_:cellForRowAt:) 和 tableView(_:willDisplay:forRowAt:) 應該只會呼叫 2 次，但它卻呼叫了 10 次，傳入的參數，indexPath 的 row 依序為 0 ~ 9，分別對應第一個到第 10 個 cell。

會有這樣的現象，是因為 estimatedRowHeight 預設被設為UITableViewAutomaticDimension，table 一開始會先預測 cell 高度，用此高度算出一開始畫面要顯示的 cell 數量。因此如果它預測畫面上可以顯示 10 個 cell，就會呼叫 10 次，即使實際上畫面只看到 2 個 cell。

這個現象看來問題不大，但當我們想實作滑動到最後一個 cell 抓取網路上的資料，顯示更多時，卻會發生問題。因此一般我們會在 function tableView(_:willDisplay:forRowAt:) 利用參數 indexParth 做判斷。如果 indexPath.row 等於資料 array 的數量減 1 時，即代表已滑到最後一個 cell，可以連到網路上抓資料。

func tableView(_ tableView: UITableView, willDisplay cell: UITableViewCell, forRowAt indexPath: IndexPath) {
}

但若發生剛剛的現象，我們其實還沒滑到最後一個 cell，但卻因為 tableView(_:willDisplay:forRowAt:) 被呼叫，它的 indexPath.row 等於 array 的數量減 1，讓我們誤以為已滑到最後一個 cell，該抓取更多資料了。

解法

將 estimatedRowHeight 設為 cell 最有可能的高度，比方將它設為 200，table 將用 200 推算一開始要顯示的 cell 數量，如此算出來的數值會比較準確，降低誤會已滑到最後一個 cell 的機率。

tableView.estimatedRowHeight = 200
